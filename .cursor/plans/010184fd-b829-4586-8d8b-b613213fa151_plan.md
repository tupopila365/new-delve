# Fast Publish: Eager Upload + Deferred Bake

**Goal:** Make Share feel instant by uploading during edit time, parallelizing carousel uploads, and eventually moving ffmpeg bake off the request path.

**Constraint:** No Redis/Celery today (`Procfile` = web only). Phase 1 ships without new infra.

---

## Recommended approach (3 phases)

| Phase | What users feel | Infra |
|-------|-----------------|-------|
| **1 — ship first** | Upload while editing; Share mostly waits on POST; optimistic feed card | None |
| **2** | Grade/overlay videos also Share in ~2s; bake finishes in background | Heroku Scheduler (+ optional daemon thread) |
| **3** | Flaky mobile + huge files survive; less ffmpeg via CDN transforms | Optional worker dyno |

---

## Phase 1 — Eager upload + parallel + optimistic UI

### Why this first
Today Share does **upload + bake** serially. Trim remux is already skipped; images are still fully canvas-baked; videos with filters still **sync-bake** in `perform_create`. Eager upload alone removes the biggest Share wait without workers.

### Changes

1. **Slide upload state** in `SocialCreateComposer.tsx`
   - Add `upload: { status, publicId, secureUrl, editFingerprint, error? }` to `SlideState`
   - Fingerprint = hash of file + crop/filter/overlays/trim so we only re-upload when edits change

2. **NEW** `frontend/src/components/create/socialMediaApi.ts` (mirror `highlightMediaApi.ts`)
   - `ensureSlideUploaded(slide)` — compress if needed → Cloudinary direct upload → store `public_id`
   - `uploadSlidesParallel(slides)` — `Promise.allSettled` (cap concurrency at 3)
   - Debounced (~500ms) trigger on edit settle / slide switch / add media

3. **Slim Share** in `SocialCreateComposer.tsx` `publish.mutationFn`
   - Await pending uploads (usually no-op)
   - POST metadata + `*_public_id` + trim + grade/overlay PNG only
   - No per-slide sequential upload loop at Share time

4. **Smart compress** in `delversVideoUtils.ts`
   - Keep >2MB compress path
   - Skip compress when Cloudinary direct is on **and** probe shows already-web-friendly (H.264, ≤1080p) — avoid remux for “good enough” phone files

5. **Optimistic UI**
   - `socialCache.ts`: `prependOptimisticDelversPost` / `reconcileOptimisticPost`
   - `SocialCreateComposer`: `onMutate` prepend temp post → navigate → reconcile on success (pattern from `DelversSocial.tsx` like/save)
   - Soft slide-thumb indicators: uploading / ready / error + retry

6. **Backend (minimal)**
   - Optional batch sign on `MediaSignView` (one request for N slides) in `cloudinary_upload.py` + `views.py`
   - Optional local-dev `POST /api/social/media/upload/` if direct Cloudinary is off (mirror highlights)
   - **No migration**

### UX
| Path | After Phase 1 |
|------|----------------|
| Trim-only video | Share ≈ POST (~1–3s) + optimistic card |
| Image | Bake during idle edit; Share ≈ POST |
| Grade/overlay video | Share waits on **ffmpeg only** (upload already done) |
| 5-slide carousel | Uploads overlap during editing |

## Phase 2 — Never bake on the hot path

### Approach (no Celery)
1. Add `processing_status` (`ready` | `processing` | `failed`) on `Post` + `PostMedia`
2. Persist `edit_grade` JSON + overlay PNG asset on create
3. `perform_create`: attach public_ids → set `processing` → **return 201 immediately** (no `bake_video_field` in-request)
4. Bake via `manage.py bake_pending_posts` (Heroku Scheduler 1–5 min) + best-effort `on_commit` daemon thread for immediacy
5. Feed shows “Finalizing video…”; poll/`GET` until `ready`; reconcile optimistic card
6. Images stay client-baked (WYSIWYG draw/stickers)

### Critical files
- `backend/social/models.py` + migration
- `backend/social/views.py` (`perform_create`, `_apply_video_edits`)
- `backend/social/serializers.py` (`processing_status` on feed)
- `backend/social/management/commands/bake_pending_posts.py` *(new)*
- Feed card UI (`DelversSocial` / post card)

### Verify
- Create with grade → 201 + `processing`; no sync bake mock call
- Command flips row to `ready`
- Failure → `failed` + retry path

## Phase 3 — Chunked upload + richer CDN transforms

1. **Chunked Cloudinary upload** in `cloudinaryUpload.ts` for videos >~10MB (`X-Unique-Upload-Id`)
2. **Optional** Cloudinary delivery for approximate grade/overlay (`cloudinary_media.py` + feature flag); keep ffmpeg for exact WYSIWYG
3. Orphan eager-upload cleanup cron
4. Worker dyno only if Phase 2 backlog is too slow

---

## Explicit non-goals (for now)
- Celery/RQ/Redis (unless Phase 2 latency forces it)
- Client-side video remux for trim (already correct; don’t regress)
- Server image re-bake (client canvas remains source of truth)
